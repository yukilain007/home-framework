# HOME 日常使用：如何保存、审核和更新你的 AI 上下文

> 不保存所有对话，只保留未来仍会影响协作的信息。

第一次使用 HOME，可以先阅读：

[HOME 零技术用户安装与使用指南](zero-tech-user-guide.zh-CN.md)

这份指南适合已经完成安装和第一次初始化的用户。你不需要理解 YAML、schema 或 `core / current / candidate` 这些内部名称；你只需要判断：**这条信息以后还会不会影响我和 AI 的协作？如果会，应该留多久、给谁看？**

## 先知道 HOME 不做什么

HOME 当前不会自动读取你的全部聊天历史，也不会自动判断哪句话应成为长期事实。

它没有消费者 GUI、交互式审核界面、自动每日提醒或聊天平台自动同步功能。

当前实际 CLI 以以下命令为准：

- `home init`：创建一个可运行的示例工作区；正式使用前，需要审核并替换其中的示例内容。
- `home validate`：检查工作区中的结构和引用；
- `home build`：根据已确认、可用的信息生成一份 Context Handoff；
- `home doctor`：检查工作区、有效期、导出、敏感信息与 Git 卫生状况。

本文说的“AI 整理”“AI 建议”和“周期审核”，是你借助**具备电脑、文件和终端操作权限的 AI**完成的工作流，不是 HOME CLI 已经提供的自动审核能力。

HOME 不承诺永久记忆或不同窗口之间的意识连续。AI 的推测也不能直接成为你的事实。

## 今天遇到一条信息怎么办？

先不用急着“记下来”。下面五种去向只回答一个问题：**这条信息要保存多久？**

| 去向 | 适合什么 | 例子 | 接下来怎么做 |
| --- | --- | --- | --- |
| 1. 不保存 | 一次性情绪、闲聊、与未来协作无关的信息，或不想让任何 AI 使用的私人内容 | “我今天有点烦。” | 不写进 HOME；需要时只在当次对话里说。 |
| 2. 仅用于当次任务 | 只为眼前任务提供的短暂背景 | “这篇文章今天要改成 800 字。” | 只放进当前任务提示或临时笔记；任务结束后不作为 HOME 上下文复用。 |
| 3. 保留到当前项目结束 | 仍会影响这个项目、但项目结束后大概率失效的状态 | “本周先完成采访稿，不开始下一篇。” | 在本地项目的当前状态记录中保留，并写清楚何时应复查或结束。 |
| 4. 先列为候选 | 看起来未来可能有用，但你还没确认是否准确、稳定或可共享 | “我可能更喜欢这种写作节奏。” | 单独列为候选；它不能进入最终 Context Handoff。 |
| 5. 长期保留 | 你已经确认、长期有效、未来多个任务都会用到的信息 | “所有对外 Handoff 都应先展示共享范围。” | 在你明确批准后，写入经过审阅的长期信息；定期复查是否仍适用。 |

## 再决定：谁可以看到它

保存期限不等于共享许可。无论一条信息是“当次任务”还是“长期保留”，还要单独决定它能否进入外部 AI 的 Handoff。

| 共享范围 | 含义 |
| --- | --- |
| 可以进入外部 AI Handoff | 在用户批准的任务和范围内，可以交给外部 AI。 |
| 只保留在本地 | 可以保存在 HOME 工作区，但不能进入外部 AI Handoff。 |
| 每次使用前询问 | 不能自动复用；每次进入 Handoff 前都要重新确认。 |
| 永远不进入 Handoff | 不得写入任何可交付给 AI 的上下文文件。 |

**保存期限和共享范围是两个不同的决定，需要分别确认。**

例如：

- `长期保留 + 只保留在本地`：这条信息对你长期有用，但不应交给外部 AI。
- `保留到项目结束 + 每次使用前询问`：它在这个项目期间仍有用，但每次要不要交给某个 AI 都必须重新决定。

关键不是“多保存”，而是**不把短期状态、AI 的猜测或一时情绪，误写成你长期稳定的事实。**

## 一个简单判断法

遇到一条信息时，可以问自己四个问题：

1. 三个月后，它还会影响我和 AI 的协作吗？
2. 它是我明确说过的事实，还是 AI 根据语气做出的推测？
3. 我愿意让哪个 AI、在什么任务中看到它？
4. 如果明天它变了，我希望怎样更新或撤回？

如果第 2 题答不出来，先选“候选”。如果第 3 题不愿回答，先决定保存期限，再把共享范围设为“只保留在本地”或“永远不进入 Handoff”。

## 日常维护：先看变化，再决定是否写入

HOME 的日常维护不需要每天做。更实用的顺序是：

1. 你完成一个阶段、改变一个项目方向，或发现旧信息不再准确；
2. 请协作 AI 先展示“它建议改变什么、为什么、哪些内容可能受影响”；
3. 你确认要不要改、应落入五种保存期限中的哪一种，以及共享范围是什么；
4. AI 才能修改经过你批准的本地信息，并运行相应检查；
5. 如果要给新窗口或新工具使用，再生成并审阅一份新的 Handoff。

AI 可以帮你发现重复、提醒某条状态可能过期、把你已说清楚的内容整理成草稿；但它不能因为“看起来合理”就替你批准，也不能把候选信息当作已确认事实。

### 事件触发优先

以下情况比“定期打扫”更值得立刻检查：

- 一个项目阶段完成、暂停或取消；
- 目标、合作方式、隐私边界或共享范围改变；
- 你发现旧 Handoff 里有过期或不准确的内容；
- 你准备把同一项目交给新的窗口、模型或工具；
- 你说出了“这已经不再适合我”之类的变化。

### 每周与每月检查只是建议

如果你想建立轻量习惯，可以参考：

- **每周 5 分钟**：只看当前项目状态是否过期、候选是否堆积；
- **每月 15 分钟**：看长期保留的信息是否重复、冲突或不再适用；
- **项目结束时**：把“保留到当前项目结束”的内容删除、归档，或在你确认后提升为长期信息。

这些是你和协作 AI 可以采用的工作习惯，不是 HOME CLI 的自动提醒、后台任务或日历功能。

## 什么时候运行 CLI？

你不需要每次聊天都运行命令。一般来说：

| 你刚做完什么 | 建议做什么 |
| --- | --- |
| 第一次建立工作区 | 使用 `home init`；它不是日常“保存记忆”的命令。 |
| 修改了已批准的本地信息 | 运行 `home validate`，先确认结构与引用没有问题。 |
| 准备交给新的 AI | 运行 `home build`，生成针对当前任务的 Context Handoff。 |
| 想检查过期状态、导出或敏感信息风险 | 运行 `home doctor`。 |

在让协作 AI 执行前，请它先用当前版本的 `--help` 核对命令参数。不要凭这份指南猜测路径、选项或文件格式。

## 更新 Handoff 前的四条规则

1. **候选不进 Handoff。** 未确认的候选信息不能被 `home build` 当作最终上下文。
2. **当前消息优先。** 你在当前对话中明确提出的新要求，高于旧 Handoff 中的记录。旧文件可以帮助理解，不能替你定义今天的决定。
3. **旧 Handoff 不是长期事实。** 它只是某次任务、某个日期下生成的交接文件；变化后应重新审阅并重新生成。
4. **最终批准才可交付。** 先把生成结果标记为“待用户批准”；只有你看过共享范围并明确批准后的 Handoff，才可以交给新的窗口、模型或工具。

批准某一份 Handoff，不等于同意以后自动共享同样内容。每次任务仍可以缩小、修改或撤回范围。

## 可复制的日常维护提示词

把下面这段交给具备电脑、文件和终端操作权限的 AI。它不会让 HOME 自动审核；它要求 AI 先展示建议、等待你确认，再做实际操作。

```text
请协助我做一次 HOME 日常上下文维护。先说明你是否具备真实的文件和终端操作权限；如果没有，不要假装已经读取或修改文件，只给我一个可审阅的维护清单。

在读取任何本地私人文件前，先告诉我你准备读取哪些文件、每个文件的用途，以及你的执行环境是否属于云端服务。等待我同意后再继续；不要读取无关文件。

请先只展示建议，不修改任何内容。把每一条信息分到以下五种去向之一：
1. 不保存；
2. 仅用于当次任务；
3. 保留到当前项目结束；
4. 先列为候选；
5. 长期保留。

再为每一条信息单独标注共享范围：可以进入外部 AI Handoff、只保留在本地、每次使用前询问，或永远不进入 Handoff。保存期限和共享范围必须分别等待我确认。

对每条建议说明：它来自哪里、为什么可能有用、是否已经被我明确确认、保存期限、共享范围，以及它是否会进入下一份 Context Handoff。

不要把你的推测写成我的事实；不要把候选信息放入 Handoff；不要自动把短期项目状态提升为长期信息；不要主动把文件上传到额外的第三方服务、公开仓库或共享链接。开始前请说明你的执行环境是否属于云端服务，以及你读取的内容是否可能进入服务商的处理范围；不要写入 API key、密码、证件号码、银行卡信息、精确住址或其他高敏感数据。

等我逐项确认后，再修改获准的本地信息。修改后，用当前 CLI help 核对参数，并在适用时运行 home validate、home build、home doctor。最后向我展示：改了什么、哪些内容没有改、这份 Handoff 会看到什么、不会看到什么，以及生成的本地文件路径。

当前对话中的新要求优先于旧 Handoff。只有我明确批准最终 Handoff 后，才能把它交给新的 AI 工具或窗口。
```

## 隐私与退出权

- 不需要保存全部聊天，才能让未来协作更顺畅。
- 不确定、敏感、属于他人或不想共享的内容，默认不要进入 Handoff。
- HOME 是 local-first：工作区由用户本地管理；如果协作 AI 是云端服务，它读取的内容可能进入服务商处理范围，在提供私人内容前应单独确认。
- 你可以要求删除、缩小共享范围、把一条信息降为候选，或直接停止维护，不需要解释理由。

HOME 的价值不是让 AI “知道得越多越好”，而是让你清楚：**它这一次依据了什么，以及哪些内容仍然由你保留。**
